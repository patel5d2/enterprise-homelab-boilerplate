# Security Policy

## Supported Versions

| Version | Supported          |
| ------- | ------------------ |
| 2.x     | :white_check_mark: |
| < 2.0   | :x:                |

## Reporting a Vulnerability

Please report security vulnerabilities privately via
[GitHub Security Advisories](https://github.com/patel5d2/enterprise-homelab-boilerplate/security/advisories/new).
Do not open a public issue for security problems.

You can expect an initial response within 7 days.

## Security Measures in This Project

- **Secrets management**: `.env` files and their backups are written with
  owner-only permissions (`0600`) and are excluded from version control.
- **Password hashing**: HTTP basic-auth credentials are hashed with bcrypt
  (SHA-512 crypt fallback on older platforms). Weak hash fallbacks are
  rejected at runtime.
- **Generated containers**: services generated by `labctl` default to
  `no-new-privileges`, log rotation, and health checks.
- **TLS verification**: health checks verify TLS certificates by default;
  verification is only relaxed when staging certificates are explicitly
  configured.
- **CI scanning**: every push and pull request runs `bandit` (static
  security analysis) and `pip-audit` (known-vulnerability scan).

## Hardening Recommendations for Deployments

- Never commit `.env`, certificates, or private keys. The provided
  `.gitignore` covers these, but verify before pushing.
- Use strong, generated passwords (`labctl` generates 32-character
  passwords with full character classes by default).
- Keep service images pinned and updated; subscribe to upstream advisories
  for the services you enable.
- Restrict access to the Docker socket. Only Traefik and Uptime Kuma mount
  it read-only by default; Jenkins requires read-write access — only enable
  it if you understand the implications (socket access is root-equivalent).
